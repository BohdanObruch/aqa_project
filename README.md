# AQA Test Project

Тестовый API-проект для проверки подхода к проектированию AQA-решения, написанию E2E-тестов и обоснованию архитектуры.

## Цель

Проект покрывает `https://jsonplaceholder.typicode.com` и демонстрирует:

- расширяемую структуру под рост количества API и сценариев;
- отделение тестов от низкоуровневой работы с HTTP;
- читаемые E2E-сценарии с пере используемыми проверками;
- готовность к работе в условиях микросервисной архитектуры SUT.

## Стек

- Python 3.12+
- Pytest
- Requests
- Allure

## Структура проекта

```text
aqa_test/
|-- src/
|   |-- config/         # конфигурация окружения
|   |-- core/           # базовый HTTP-клиент, парсеры, общие проверки
|   |-- data/           # тестовые данные и payloads
|   |-- models/         # модели доменных сущностей
|   |-- services/
|       |-- posts/      # клиент сервиса/домена posts
|       |-- users/      # клиент сервиса/домена users
|       |-- comments/   # клиент сервиса/домена comments
|-- tests/
|   |-- unit/           # unit-тесты парсеров и изолированной логики
|   |-- component/      # component-тесты клиентов без реального HTTP
|   |-- e2e/            # end-to-end API сценарии
|   |-- conftest.py     # фикстуры клиентов
|-- pytest.ini
|-- requirements.txt
|-- README.md
```

## Почему архитектура сделана с учетом микросервисного SUT

Хотя `JSONPlaceholder` выглядит как единый публичный API, тестовый проект специально разделен по доменам `posts`, `users`, `comments`. Это имитирует подход, который нужен в системе с несколькими сервисами и большим количеством интеграций.

Такой вариант выбран потому что он лучше масштабируется:

- можно добавлять новые домены или сервисы без переписывания существующих тестов;
- у каждого сервиса может быть свой клиент, свои модели и свои данные;
- E2E-сценарии могут проверять цепочки вызовов между несколькими сервисами;
- проще поддерживать изоляцию, переиспользование и читаемость кода.

## Как выглядело бы решение для монолита

Если бы SUT был монолитом, структура могла бы быть проще:

```text
aqa_test/
|-- api_client.py
|-- test_data.py
|-- tests/
|   |-- test_posts.py
|   |-- test_users.py
|-- conftest.py
```

Для монолита обычно характерно:

- один общий API-клиент на все эндпоинты;
- меньше слоев абстракции;
- тесты чаще сгруппированы по функциональности внутри одного приложения;
- меньше внимания к границам между сервисами и интеграционным контрактам.

## Чем это отличается от подхода для микросервисов

На примере текущего проекта:

- для монолита можно было бы держать все запросы в одном клиенте `api_client.py`;
- для микросервисного подхода клиенты разделены по доменам в `services/posts`, `services/users`, `services/comments`;
- для монолита тест мог бы просто проверить один endpoint;
- для микросервисного подхода E2E-сценарий связывает несколько частей системы, например: получить post, затем автора этого post, затем связанные comments.


## Покрытые сценарии

Реализованы E2E-тесты:

1. Получение `post`, затем `user` автора и затем `comments` этого `post` с проверкой связности данных.
2. Создание нового `post` и проверка полей ответа.
3. Обновление существующего `post` и последующее удаление с проверкой результата.

Дополнительно, как опциональная часть задания, реализованы тесты других уровней:

1. `unit`-тест на парсинг полной схемы пользователя в модель с вложенными объектами.
2. `component`-тест на клиент `PostsClient`, который проверяет корректную сборку HTTP-запроса без реального вызова внешнего API.


## Установка

```bash
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
```

## Запуск тестов

```bash
pytest
```

Запуск только E2E-тестов:

```bash
pytest -m e2e
```

Запуск только дополнительных тестов других уровней:

```bash
pytest -m "unit or component"
```

Запуск отдельно по уровням:

```bash
pytest -m unit
pytest -m component
```

## Запуск Allure-отчета

Сбор результатов:

```bash
pytest
```

Результаты складываются в `allure-results` через `pytest.ini`.

### Вариант для Allure 2.x

```bash
allure serve .\allure-results
```

Или:

```bash
allure generate .\allure-results -o .\allure-report --clean
allure open .\allure-report
```

### Вариант для Allure 3.x

```bash
allure generate .\allure-results -o .\allure-report
allure open .\allure-report
```

Или:

```bash
allure open .\allure-results
```

Примечание:

- в Allure 2.x обычно используется `serve` и поддерживается `--clean` при `generate`;
- в Allure 3.x удобнее использовать `generate` и `open`, а синтаксис CLI отличается от 2.x.

## Почему такой подход удобен для развития

- Новый сервис можно добавить отдельной папкой в `src/services/`.
- Общие проверки не дублируются по тестам.
- Тестовые данные вынесены отдельно.
- Тесты читаются как бизнес-сценарии, а не как технические детали HTTP-запросов.

Дополнительное преимущество такого решения в том, что проект уже поддерживает несколько уровней тестирования:

- `unit` для быстрой проверки локальной логики и преобразований данных;
- `component` для проверки клиентов и внутренних интеграций без зависимости от внешней сети;
- `e2e` для проверки пользовательских API-сценариев целиком.
